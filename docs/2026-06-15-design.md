# eda-watcher — Local 2-Panel Research Board

**Date:** 2026-06-15
**Status:** Approved design

## Problem

During research sessions, Claude Code generates Python scripts and PNG
charts (EDA, sweeps, backtests). The user cannot conveniently view these
inline — switching to an external file viewer for each image and opening
the script in an editor breaks flow. We want a single local view: code on
one side, the images it produced on the other.

This is a standalone tool, intentionally kept out of any domain repo: it
has zero domain dependencies (stdlib HTTP server + HTML/JS only) and is
reusable across any project's Claude Code session — every session writes
to one shared manifest, and this board reads it.

## Goal

A localhost-only dashboard with two panels:

- **Left:** a vertical list of entries (top) + the selected entry's code
  (bottom). An entry may have multiple scripts → sub-tabs in the code
  panel.
- **Right:** the images that the selected entry produced.

Entries come from wherever the current session writes — `tmp/`, a git
worktree, or a repo. Paths are not fixed; they are recorded as **absolute
paths** in a manifest by Claude when the artifacts are created.

## Non-Goals

- No remote hosting (localhost only).
- **Read-only.** The dashboard never writes/modifies the manifest, never
  touches any file on disk. No delete feature, no edit, no POST/DELETE
  routes. Manifest is written ONLY by Claude during research.
- No auth.

## Architecture

Three pieces.

### 1. Manifest — single source of truth

Path: `~/.claude/eda-watcher/manifest.json`

JSON array of entries, each:

```json
{
  "id": "rsi-ma-sweep-4yr",
  "title": "RSI x MA sweep (4yr)",
  "created": "2026-06-15T10:30:00",
  "scripts": ["/abs/path/to/sweep.py", "/abs/path/to/helper.py"],
  "images": ["/abs/path/a.png", "/abs/path/b.png"],
  "note": "optional one-liner"
}
```

- `id`: unique slug. Appending an entry with an existing `id` replaces it
  (idempotent re-run).
- `scripts`: array — an entry may produce/use multiple `.py` files. The
  code panel shows one sub-tab per script.
- All paths are **absolute** — session/worktree/tmp agnostic.
- Claude appends/updates an entry when it generates research code + plots.
  The board only ever **reads** this file.
- The manifest directory (`~/.claude/eda-watcher/`) is created on first
  write by Claude if missing. The server treats a missing manifest as `[]`.

### 2. Server — `serve.py`

Python stdlib `http.server` only. Zero third-party deps. **GET-only** — no
route mutates anything.

Routes:

| Route | Behavior |
|-------|----------|
| `GET /` | serve `index.html` |
| `GET /app.js` | serve `app.js` |
| `GET /api/manifest` | return manifest JSON, **newest first** (sorted by `created` desc). Empty array if manifest missing. |
| `GET /api/file?path=<abs>` | serve the file at `<abs>` — **only if that exact path appears in the manifest** (allowlist). Script files served as `text/plain; charset=utf-8`; images by extension-derived MIME. |

Security:
- GET-only; no write/delete routes exist, so the server structurally
  cannot modify the manifest or any file.
- `/api/file` rejects any path not present in the manifest's `scripts` /
  `images` fields with **HTTP 403** — prevents arbitrary filesystem reads
  via the localhost server.
- Allowlisted but missing-on-disk file → **HTTP 404**.

Run: `python serve.py` → binds `127.0.0.1:8765`. Port/host are module
constants overridable via `--port` / `--host` argparse args.

### 3. Frontend — `index.html` + `app.js`

Vanilla JS, no build step.

Layout (CSS flex, two columns):

- **Left column:**
  - Entry list at top (title + created time, newest first). Clicking an
    entry selects it.
  - Selected entry's **code** below. If the entry has multiple scripts,
    show one sub-tab per script (filename label); clicking a sub-tab shows
    that script's code. Syntax-highlighted via highlight.js (CDN); falls
    back to plain `<pre>` if CDN unavailable.
- **Right column:** selected entry's **images**, stacked vertically. Click
  an image → open full-size in a new tab (`/api/file?path=`).

Behavior:

- On load and every ~3s, poll `GET /api/manifest`. New entries appear
  without a manual reload. Current selection (entry id + active script
  sub-tab) is preserved across polls.
- Code fetched via `GET /api/file?path=<script>`; images via
  `<img src="/api/file?path=<image>">`.

## Data Flow

```
Claude runs research
  -> writes sweep.py + a.png, b.png  (anywhere: tmp/worktree/repo)
  -> appends/updates manifest entry (abs paths, scripts[] + images[])
Browser polls /api/manifest (3s)
  -> new entry appears in left list
User clicks entry
  -> left: code (script sub-tabs)    right: images
```

## Error Handling

| Condition | Behavior |
|-----------|----------|
| Manifest file missing | `/api/manifest` returns `[]`; UI shows "no entries yet". |
| Manifest malformed JSON | `/api/manifest` returns `500` with error text; UI shows error banner, keeps last good state. |
| Entry's script/image path missing on disk (e.g. tmp auto-cleaned) | `/api/file` returns 404; UI shows "file not found" placeholder for that item — does not crash the view. No auto-removal from manifest. |
| Requested path not in manifest | `/api/file` returns 403. |

## Testing

`serve.py` logic unit-tested in `test_serve.py` (pytest). Cases via
`@pytest.mark.parametrize`:

- manifest load: missing file → `[]`; valid → sorted newest-first;
  malformed → 500 path.
- `/api/file` allowlist: path in manifest → served; path not in manifest
  → 403; allowlisted but missing on disk → 404.
- MIME selection: `.py` → `text/plain`; `.png` → `image/png`.

Frontend verified manually (load board, generate a sample entry, confirm
it appears and renders, multi-script sub-tabs switch).
